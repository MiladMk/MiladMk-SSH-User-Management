## دربارهٔ x-net (که فرستادی)
پروژهٔ x-net یک پروژهٔ کاملاً جدا با معماری متفاوت (Go + React) و **لایسنس اختصاصی و بسته** است؛ در README خودش نوشته «سورس‌کد بدون حق دسترسی، تغییر یا مهندسی معکوس». پس نه سورس محاسبهٔ ترافیکش در دسترس است و نه از نظر قانونی مجاز به استفاده از کدش هستیم. برای همین از x-net استفاده نکردم — و اصلاً نیازی نبود: باگ ترافیک را با روش استاندارد و مستقل حل کردم.

## قدم‌به‌قدم بارگذاری (هیچ‌چیز هنوز بالا نرفته)

### گام ۱ — دانلود و آماده‌سازی
فایل **`MiladMk-repo-ready-for-MiladMk.zip`** را دانلود و روی کامپیوترت **Extract** کن. این پوشه محتوای کامل ریپوی توست.

### گام ۲ — ساخت ریپو (اگر هنوز نساخته‌ای)
1. برو به github.com و لاگین کن.
2. بالا سمت راست → **+** → **New repository**.
3. اسم: `MiladMk-SSH-User-Management`  (همین اسم چون لینک‌ها بر این اساس تنظیم شده‌اند).
4. Public را انتخاب کن، **بدون** تیک README/gitignore/license.
5. **Create repository**.

### گام ۳ — آپلود فایل‌ها
1. در صفحهٔ ریپوی خالی، روی **uploading an existing file** کلیک کن (یا Add file → Upload files).
2. **همهٔ محتوای داخل پوشهٔ extract‌شده** را (نه خود پوشه — بلکه فایل‌ها و پوشه‌های داخلش مثل `install.sh`, `Web Panel`, `nethogs-json`, `dropbear`, `OT` و...) بکش و رها کن.
   - اگر گیت‌هاب اجازهٔ کشیدن پوشه نداد، فایل‌ها را در چند مرحله آپلود کن، یا از Git Desktop استفاده کن.
3. پایین صفحه پیام commit بنویس (مثلاً `initial v4.3`) → **Commit changes**.

### گام ۴ — ساخت Release (بسیار مهم)
1. سمت راست صفحهٔ ریپو → **Releases** → **Create a new release** (یا Draft a new release).
2. در **Choose a tag** دقیقاً بنویس: `v4-0` و روی «Create new tag» بزن.
3. عنوان: مثلاً `v4.0`.
4. فایل **`v4-0-release-asset-debranded.zip`** را در قسمت **Attach binaries** بکش و رها کن.
5. **Publish release**.

### گام ۵ — نصب روی سرور (Ubuntu 24 یا 26 پیشنهادی)
روی یک سرور تازه به‌عنوان root:
```bash
bash <(curl -Ls https://raw.githubusercontent.com/MiladMk/MiladMk-SSH-User-Management/main/install.sh --ipv4)
```
- اگر ریپوت را `main` ساختی نه `master`، در همهٔ لینک‌ها `master` را به `main` عوض کن (یا برنچ را به master تغییر بده). می‌توانم نسخهٔ main هم برایت بسازم اگر خواستی.

### گام ۶ — بعد از نصب
- دستور مدیریت: `miladmk` (یا `xpanel`).
- چرخش IP: تنظیمات ← IP Rotate.
- تست ترافیک: چند کاربر بساز، کمی مصرف کن، بعد ~۱۰ دقیقه ببین مصرف در پنل درست بالا می‌رود.

## Changelog v4.3

### قطع کامل از پروژهٔ قبلی
- حذف کامل XGuard و لایسنس (از نسخهٔ قبل).
- **حذف حتی fallbackهای xpanel-cp**: nethogs و dropbear حالا فقط از ریپوی خودت نصب می‌شوند.
- `nethogs-json/install.sh` و `dropbear/install.sh` را از صفر و مستقل نوشتم (داخل ریپو هستند).
- تأیید نهایی: **صفر مورد** xpanel-cp / xpanel.pro / Alirezad در کل کد.

### رفع باگ ترافیک
- حذف ضریب دلخواه `TRAFFIC_BASE` (عدد ۱۲ که روی سرورهای مختلف نتیجهٔ غلط می‌داد).
- تبدیل استاندارد: nethogs مصرف را بر حسب KB می‌دهد → تقسیم بر ۱۰۲۴ = MB (واحدی که دیتابیس ذخیره می‌کند). مستقل از سرور.
- روشن/خاموش کردن محاسبهٔ ترافیک همچنان از تنظیمات ← عمومی (سوییچ `CRON_TRAFFIC`) کار می‌کند تا در صورت نیاز بار سرور کم شود.

### از نسخه‌های قبل (همچنان داخل)
- چرخش IP کلودفلر مستقل، هاست سفارشی، NPV Tunnel، برند MiladMk، دستور miladmk، سازگاری Ubuntu 24/26 + Debian، رفع php-fpm sandbox.

## دو نکتهٔ صادقانه (مهم)

1. **nethogs-json که نوشتم بازسازی خودم است.** بر اساس فرمتی که پنل از `out.json` می‌خواند و خروجی استاندارد nethogs نوشتمش. منطقش درست است، اما نگاشت دقیق واحدها (TX/RX و KB) را حتماً روی سرورت تست کن: چند کاربر بساز، مصرف کن، ببین عدد پنل با مصرف واقعی می‌خواند. اگر نخواند، خروجی `cat /var/www/html/app/storage/out.json | tail -1` را برایم بفرست تا دقیق کالیبره کنم.

2. **چون سرور/PHP اینجا نبود، نتوانستم زنده اجرا/لینت کنم.** ساختار bash، JSON و PHP را با دقت چک کردم و از الگوهای خود پروژه پیروی کردم. اولین نصب را حتماً روی سرور تست انجام بده و هر خطایی دیدی بفرست.
